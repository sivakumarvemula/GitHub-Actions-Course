Chapter 1: GitHub Actions Fundamentals

GitHub Actions is a powerful automation platform integrated directly into GitHub repositories. It allows you to automate your software development workflows, from simple tasks like running tests to complex CI/CD pipelines.

What is GitHub Actions?

GitHub Actions is GitHub's built-in continuous integration and continuous delivery (CI/CD) solution that allows you to automate your build, test, and deployment pipeline directly from your GitHub repository.

Unlike external CI/CD tools, GitHub Actions is fully integrated with GitHub, making it seamless to automate workflows based on GitHub events like pushes, pull requests, releases, and more.

Core Components

Workflows

Automated procedures that you add to your repository, defined in YAML files in the .github/workflows directory.

Events

Specific activities that trigger a workflow run, such as a push, pull request, or scheduled time.

Jobs

A set of steps that execute on the same runner. Jobs can run in parallel or sequentially depending on configuration.

Steps

Individual tasks within a job that can run commands, setup tasks, or run an action.

Actions

Reusable units of code that can be used in workflows. They can be created by you or sourced from the GitHub Marketplace.

Runners

Servers that run your workflows when they're triggered. GitHub provides Linux, Windows, and macOS runners, or you can host your own.

Key Benefits of GitHub Actions

  • Seamless Integration: Built directly into GitHub, eliminating the need for external services.
  • Marketplace of Actions: Access to thousands of pre-built actions for common tasks.
  • Matrix Builds: Test across multiple operating systems and runtime versions easily.
  • Community-Powered: Benefit from a large community creating and sharing actions.
  • Free for Public Repositories: Generous free tier for open-source projects.
  • Self-Hosted Runners: Option to use your own infrastructure for specific needs.

Important Note

GitHub Actions follows specific conventions for workflow files. All workflow files must be stored in the .github/workflows directory of your repository and must use YAML syntax with either a .yml or .yaml file extension.

Chapter 2: Creating Your First Workflow

In this chapter, we'll learn how to create and configure GitHub Actions workflows. We'll start with a simple workflow and gradually build more complex ones.

Workflow File Structure

GitHub Actions workflows are defined in YAML files located in the .github/workflows directory of your repository. Each workflow file contains the following key elements: 

name: My First Workflow

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
      - name: Install dependencies
        run: npm ci
      - name: Run tests
        run: npm test

name

The name of your workflow that appears in the GitHub Actions tab. This is optional but recommended.

on

Specifies the events that trigger the workflow. In the example above, the workflow runs when code is pushed to the main branch or when a pull request is created against the main branch.

jobs

A workflow consists of one or more jobs. Jobs run in parallel by default unless you specify dependencies.

runs-on

Specifies the type of runner to use for the job. GitHub provides runners for Ubuntu Linux, Windows, and macOS.

steps

A job contains a sequence of steps. Steps can run commands, set up tasks, or run an action.

uses

Specifies an action to run as part of a step. In the example, we're using the checkout action to fetch the repository code.

with

Provides input parameters for an action.

run

Runs command-line programs using the operating system's shell.

Creating Your First Workflow

Let's create a simple workflow that runs tests for a Node.js application:

  1. Create the workflow directory

    mkdir -p .github/workflows

  2. Create a workflow file

    touch .github/workflows/node-tests.yml

  3. Add the following content to the file:

    name: Node.js Tests

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test

  4. Commit and push the workflow file to your repository

    git add .github/workflows/node-tests.yml
    git commit -m "Add Node.js test workflow"
    git push

Once you push this file to your repository, GitHub Actions will automatically run the workflow whenever code is pushed to the main branch or when a pull request is created against the main branch.

Understanding Workflow Events

Events are specific activities that trigger a workflow run. GitHub provides many event types, including:

push

Triggered when code is pushed to the repository.

on:
  push:
    branches: [ main ]
    paths:
      - 'src/**'

pull_request

Triggered when a pull request is opened, synchronized, or reopened.

on:
  pull_request:
    types: [opened, synchronize]
    branches: [ main ]

schedule

Triggered at scheduled times using cron syntax.

on:
  schedule:
    - cron: '0 0 * * *' # Runs at midnight every day

workflow_dispatch

Allows manual triggering of a workflow from the GitHub UI.

on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy to'
        required: true
        default: 'staging'

Pro Tip

You can combine multiple events in a single workflow file. This is useful when you want the same workflow to run for different events.

Exercise: Create a Multi-Job Workflow

Now let's create a more complex workflow with multiple jobs that depend on each other: 

name: Build and Deploy

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Build
        run: npm run build
        
      - name: Upload build artifacts
        uses: actions/upload-artifact@v3
        with:
          name: build-files
          path: build/
  
  test:
    needs: build
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test
  
  deploy:
    needs: [build, test]
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Download build artifacts
        uses: actions/download-artifact@v3
        with:
          name: build-files
          path: build/
          
      - name: Deploy to production
        run: |
          echo "Deploying to production..."
          # Add your deployment commands here

Key points about this workflow:

  • It has three jobs: build, test, and deploy.
  • The test job depends on the build job (specified by needs: build).
  • The deploy job depends on both the build and test jobs.
  • We use the actions/upload-artifact and actions/download-artifact actions to share files between jobs.

Chapter 3: Events, Jobs, and Runners

In this chapter, we'll dive deeper into the core components of GitHub Actions: events, jobs, and runners. Understanding these components is essential for creating effective workflows.

Events in Detail

Events are specific activities that trigger a workflow run. GitHub provides a wide range of events that you can use to trigger your workflows.

Common Event Types

  • push: Triggered when commits are pushed to a repository branch.
  • pull_request: Triggered when a pull request is opened, updated, or closed.
  • schedule: Triggered at scheduled times using cron syntax.
  • workflow_dispatch: Allows manual triggering of a workflow from the GitHub UI.
  • repository_dispatch: Triggered by a custom webhook event.
  • release: Triggered when a release is created, edited, or deleted.
  • issues: Triggered when an issue is opened, edited, or closed.
  • issue_comment: Triggered when a comment on an issue or pull request is created, edited, or deleted.

Event Filtering

on:
  push:
    branches:
      - main
      - 'releases/**'
    paths:
      - '**.js'
      - 'package.json'
    tags:
      - v1.*
  pull_request:
    branches:
      - main
    paths-ignore:
      - 'docs/**'
      - '**.md'

In the example above:

  • The workflow runs on pushes to the main branch, any branch in the releases directory, and any tag that matches the pattern v1.*.
  • It only runs when JavaScript files or package.json are modified.
  • For pull requests, it only runs when the target branch is main and the changes don't only affect documentation files.

Jobs in Detail

Jobs are sets of steps that execute on the same runner. A workflow can have multiple jobs that run in parallel by default.

Job Configuration

jobs:
  build:
    name: Build Application
    runs-on: ubuntu-latest
    timeout-minutes: 10
    env:
      NODE_ENV: production
    
    steps:
      # Job steps here
  
  test:
    name: Run Tests
    needs: build
    runs-on: ubuntu-latest
    if: github.event_name == 'push'
    
    strategy:
      matrix:
        node-version: [14, 16, 18]
    
    steps:
      # Job steps here

Key job configuration options:

  • name: A display name for the job (optional).
  • runs-on: The type of runner to use for the job.
  • needs: Jobs that must complete successfully before this job will run.
  • if: Conditions that must be met for the job to run.
  • timeout-minutes: Maximum number of minutes to let a job run before GitHub automatically cancels it.
  • env: Environment variables available to all steps in the job.
  • strategy: Creates a build matrix for your jobs.

Runners in Detail

A runner is a server that runs your GitHub Actions workflows when they're triggered. GitHub provides hosted runners for Linux, Windows, and macOS, or you can host your own runners.

GitHub-Hosted Runners

  • ubuntu-latest, ubuntu-22.04, ubuntu-20.04
  • windows-latest, windows-2022, windows-2019
  • macos-latest, macos-12, macos-11

GitHub-hosted runners come with a variety of pre-installed software and tools.

Self-Hosted Runners

You can set up your own runners to use with GitHub Actions. This is useful when you need specific hardware or software configurations. 

runs-on: self-hosted

You can also use labels to target specific self-hosted runners: 

runs-on: [self-hosted, linux, x64]

Runner Specifications

GitHub-hosted runners have the following specifications:

  • 2-core CPU (x86_64)
  • 7 GB of RAM
  • 14 GB of SSD disk space
  • Ubuntu, Windows, or macOS operating system

For more demanding workflows, you might need to use larger self-hosted runners or GitHub's larger runner options.

Steps in Detail

Steps are individual tasks that can run commands or actions within a job. Steps are executed in order and are dependent on each other.

Step Configuration

steps:
  - name: Checkout code
    uses: actions/checkout@v3
  
  - name: Set up Node.js
    uses: actions/setup-node@v3
    with:
      node-version: '16'
      cache: 'npm'
  
  - name: Install dependencies
    run: npm ci
    working-directory: ./app
    env:
      NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
  
  - name: Run tests
    run: npm test
    continue-on-error: true
  
  - name: Upload test results
    if: always()
    uses: actions/upload-artifact@v3
    with:
      name: test-results
      path: test-results/

Key step configuration options:

  • name: A display name for the step (optional).
  • uses: Specifies an action to run as part of the step.
  • run: Runs command-line programs using the operating system's shell.
  • with: Provides input parameters for an action.
  • env: Environment variables available to the step.
  • working-directory: The working directory where the command is run.
  • continue-on-error: Prevents a job from failing when a step fails.
  • if: Conditions that must be met for the step to run.

Best Practice

Use descriptive names for your jobs and steps to make your workflow easier to understand. This helps with troubleshooting and makes it clear what each part of your workflow is doing.

Exercise: Create a Workflow with Event Filtering

Create a workflow that runs only when specific conditions are met: 

name: Conditional Workflow

on:
  push:
    branches:
      - main
      - 'feature/**'
    paths:
      - 'src/**'
      - 'package.json'
      - 'package-lock.json'
    tags-ignore:
      - 'v*-beta'
  pull_request:
    types: [opened, synchronize, reopened]
    branches:
      - main

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run linting
        run: npm run lint
  
  test:
    runs-on: ubuntu-latest
    needs: lint
    if: github.event_name == 'push' || github.event.pull_request.head.repo.full_name != github.repository
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test

This workflow:

  • Runs on pushes to the main branch and any feature branch, but only when changes are made to source files or package files.
  • Ignores pushes with beta version tags.
  • Runs on pull requests to the main branch.
  • Has a lint job that always runs and a test job that only runs on pushes or pull requests from forks.

Chapter 4: Working with Actions

Actions are the reusable units of code in GitHub Actions. They can perform a variety of tasks, from setting up environments to deploying applications. In this chapter, we'll explore how to use existing actions and create your own.

Types of Actions

There are three types of actions you can use in your workflows:

JavaScript Actions

Run directly on the runner and are written in JavaScript. They're fast and have access to the GitHub API. 

- uses: actions/setup-node@v3
  with:
    node-version: '16'

Docker Container Actions

Run in a Docker container and are packaged with all the dependencies they need. They're more isolated and consistent. 

- uses: docker://alpine:3.16
  with:
    args: 'echo hello world'

Composite Actions

Combine multiple steps into a single action. They're useful for reusing common sequences of steps. 

- uses: ./.github/actions/my-composite-action
  with:
    param1: 'value1'

Using Actions from the Marketplace

The GitHub Marketplace contains thousands of actions created by the community. Here are some popular actions and how to use them:

Checkout Code

- name: Checkout code
  uses: actions/checkout@v3
  with:
    # Number of commits to fetch (0 fetches all history)
    fetch-depth: 1
    # Whether to download Git-LFS files
    lfs: false
    # Whether to checkout submodules
    submodules: false

Setup Node.js

- name: Set up Node.js
  uses: actions/setup-node@v3
  with:
    node-version: '16'
    # Whether to use the node version from .nvmrc file
    node-version-file: '.nvmrc'
    # Whether to cache dependencies
    cache: 'npm'
    # Registry URL for npm
    registry-url: 'https://registry.npmjs.org'

Cache Dependencies

- name: Cache dependencies
  uses: actions/cache@v3
  with:
    # Files to cache
    path: ~/.npm
    # Cache key based on package-lock.json
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    # Fallback cache keys
    restore-keys: |
      ${{ runner.os }}-node-

Upload and Download Artifacts

- name: Upload artifact
  uses: actions/upload-artifact@v3
  with:
    name: my-artifact
    path: dist/
    # Whether to fail if no files are found
    if-no-files-found: error
    # Retention period in days
    retention-days: 5

- name: Download artifact
  uses: actions/download-artifact@v3
  with:
    name: my-artifact
    path: dist/

Create a Release

- name: Create Release
  id: create_release
  uses: actions/create-release@v1
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  with:
    tag_name: ${{ github.ref }}
    release_name: Release ${{ github.ref }}
    draft: false
    prerelease: false

Creating Your Own Actions

You can create your own actions to reuse common functionality across your workflows. Let's look at how to create a simple JavaScript action:

  1. Create a new repository for your action or use a directory in your existing repository

    mkdir -p .github/actions/hello-world

  2. Create an action.yml file to define your action

    name: 'Hello World'
description: 'Greet someone and record the time'
inputs:
  who-to-greet:
    description: 'Who to greet'
    required: true
    default: 'World'
outputs:
  time:
    description: 'The time we greeted you'
runs:
  using: 'node16'
  main: 'index.js'

  3. Create an index.js file with your action code

    const core = require('@actions/core');
const github = require('@actions/github');

try {
  // Get inputs defined in action.yml
  const nameToGreet = core.getInput('who-to-greet');
  console.log(`Hello ${nameToGreet}!`);
  
  // Get the current time
  const time = new Date().toTimeString();
  
  // Set outputs for other steps to use
  core.setOutput('time', time);
  
  // Get the JSON webhook payload for the event that triggered the workflow
  const payload = JSON.stringify(github.context.payload, null, 2);
  console.log(`The event payload: ${payload}`);
} catch (error) {
  core.setFailed(error.message);
}

  4. Create a package.json file

    {
  "name": "hello-world-action",
  "version": "1.0.0",
  "description": "A simple hello world action",
  "main": "index.js",
  "dependencies": {
    "@actions/core": "^1.10.0",
    "@actions/github": "^5.1.1"
  }
}

  5. Install dependencies

    cd .github/actions/hello-world
    npm install

  6. Use your action in a workflow

    name: Hello World Workflow

on:
  workflow_dispatch:
    inputs:
      name:
        description: 'Person to greet'
        required: true
        default: 'World'

jobs:
  hello:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Hello World Action
        uses: ./.github/actions/hello-world
        id: hello
        with:
          who-to-greet: ${{ github.event.inputs.name }}
      
      - name: Get the output time
        run: echo "The time was ${{ steps.hello.outputs.time }}"

Creating a Composite Action

Composite actions allow you to combine multiple steps into a single reusable action. Here's how to create one:

  1. Create a directory for your composite action

    mkdir -p .github/actions/setup-and-test

  2. Create an action.yml file

    name: 'Setup and Test'
description: 'Set up Node.js and run tests'
inputs:
  node-version:
    description: 'Node.js version'
    required: false
    default: '16'
runs:
  using: "composite"
  steps:
    - name: Set up Node.js
      uses: actions/setup-node@v3
      with:
        node-version: ${{ inputs.node-version }}
        cache: 'npm'
    
    - name: Install dependencies
      run: npm ci
      shell: bash
    
    - name: Run tests
      run: npm test
      shell: bash

  3. Use your composite action in a workflow

    name: Test Workflow

on:
  push:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup and run tests
        uses: ./.github/actions/setup-and-test
        with:
          node-version: '18'

Pro Tip

When creating actions, follow these best practices:

  • Keep your actions focused on a single task.
  • Document your inputs, outputs, and usage examples.
  • Version your actions using tags or branches.
  • Write tests for your actions to ensure they work as expected.

Exercise: Use Multiple Actions in a Workflow

Create a workflow that uses multiple actions from the GitHub Marketplace: 

name: CI/CD Pipeline

on:

    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

Chapter 5: CI/CD Implementation

Continuous Integration and Continuous Deployment (CI/CD) is one of the most common use cases for GitHub Actions. In this chapter, we'll explore how to implement CI/CD pipelines for different project types.

CI/CD Pipeline Overview

A typical CI/CD pipeline consists of the following stages:

Code

Test

Build

Deploy

Each stage serves a specific purpose:

  • Code: Developers write and commit code to the repository.
  • Test: The code is tested to ensure it meets quality standards and doesn't introduce regressions.
  • Build: The code is compiled, bundled, or packaged for deployment.
  • Deploy: The built artifacts are deployed to the target environment.

CI/CD for a Node.js Application

Let's create a CI/CD pipeline for a Node.js application that deploys to Heroku: 

name: Node.js CI/CD

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Lint
        run: npm run lint
      
      - name: Test
        run: npm test
      
      - name: Upload test coverage
        uses: codecov/codecov-action@v3
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
  
  build:
    needs: test
    if: github.event_name == 'push'
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Use Node.js

Chapter 6: Advanced Features

Matrix Builds

Matrix builds allow you to test your code across multiple configurations, such as different operating systems, Node.js versions, or browsers. 

name: Node.js Matrix Build

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ${{ matrix.os }}
    
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [14.x, 16.x, 18.x]
        exclude:
          # Exclude Node 14 on macOS
          - os: macos-latest
            node-version: 14.x
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Use Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v3
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test

In this example, the workflow will run 8 jobs in parallel (3 operating systems 3 Node.js versions - 1 excluded combination), testing your code across different environments.

Caching Dependencies

Caching dependencies can significantly speed up your workflows by reusing files from previous runs. 

name: Node.js with Caching

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '16'
          
      - name: Cache Node.js modules
        uses: actions/cache@v3
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-
            
      - name: Install dependencies
        run: npm ci
        
      - name: Run tests
        run: npm test

The actions/cache action caches the npm dependencies based on the contents of the package-lock.json file. If the file hasn't changed, the cached dependencies will be restored, saving download time.

Artifacts and Workflow Outputs

Artifacts allow you to share files between jobs, while outputs enable you to share data. 

name: Build and Deploy with Outputs

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      build_id: ${{ steps.build_step.outputs.build_id }}
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Build
        id: build_step
        run: |
          echo "Building application..."
          echo "build_id=$(date +%s)" >> $GITHUB_OUTPUT
          
      - name: Upload build artifacts
        uses: actions/upload-artifact@v3
        with:
          name: build-files
          path: build/
  
  deploy:
    needs: build
    runs-on: ubuntu-latest
    
    steps:
      - name: Download build artifacts
        uses: actions/download-artifact@v3
        with:
          name: build-files
          path: build/
          
      - name: Deploy with build ID
        run: |
          echo "Deploying build ${{ needs.build.outputs.build_id }}"

In this example:

  • The build job generates a build ID and sets it as an output.
  • The deploy job accesses this output using needs.build.outputs.build_id.
  • We also use artifacts to share the build files between jobs.

Chapter 7: Best Practices and Optimization

In this chapter, we'll explore best practices and optimization techniques for GitHub Actions workflows.

Workflow Optimization

Optimizing your workflows can save time, reduce costs, and improve reliability. Here are some optimization techniques:

Use Caching

Cache dependencies, build outputs, and other files to speed up your workflows. 

- name: Cache dependencies
  uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-

Optimize Checkout

Use the fetch-depth option to limit the number of commits to fetch. 

- name: Checkout code
  uses: actions/checkout@v3
  with:
    fetch-depth: 1  # Only fetch the latest commit

Use Conditional Execution

Use the if condition to skip jobs or steps that aren't needed. 

- name: Run only on main branch
  if: github.ref == 'refs/heads/main'
  run: echo "This only runs on the main branch"

Optimize Matrix Builds

Use exclude and include to fine-tune your matrix builds. 

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node-version: [14.x, 16.x, 18.x]
    exclude:
      - os: macos-latest
        node-version: 14.x
    include:
      - os: ubuntu-latest
        node-version: 20.x
        experimental: true

Use Job Outputs

Share data between jobs using outputs instead of artifacts when possible. 

jobs:
  job1:
    outputs:
      output1: ${{ steps.step1.outputs.test }}
    steps:
      - id: step1
        run: echo "test=hello" >> $GITHUB_OUTPUT
  
  job2:
    needs: job1
    steps:
      - run: echo ${{ needs.job1.outputs.output1 }}

Security Best Practices

Security is crucial when setting up GitHub Actions workflows. Here are some best practices:

Use Secrets for Sensitive Data

Never hardcode sensitive information in your workflow files. Use GitHub Secrets instead.

Pin Actions to a Specific Version

Always pin actions to a specific version (preferably a SHA) to prevent supply chain attacks. 

# Good: Pinned to a specific version
- uses: actions/checkout@v3

# Better: Pinned to a specific SHA
- uses: actions/checkout@a12a3943b4bdde767164f792f33f40b04645d846

# Bad: Not pinned to a specific version
- uses: actions/checkout@main

Limit Permissions

Limit the permissions granted to the GitHub token to only what's needed. 

permissions:
  contents: read
  issues: write
  pull-requests: write

Use Environment Protection Rules

Set up environment protection rules for sensitive environments like production.

Validate External Inputs

Always validate inputs from external sources to prevent injection attacks.

Workflow Organization

Organizing your workflows effectively can make them easier to maintain and understand:

Use Descriptive Names

Use descriptive names for your workflows, jobs, and steps to make them easier to understand.

Split Complex Workflows

Split complex workflows into smaller, more focused ones. Use reusable workflows for common functionality.

Use Comments

Add comments to your workflow files to explain complex or non-obvious parts. 

# This job runs only when a tag is pushed
jobs:
  deploy:
    # Only run this job when a tag is pushed
    if: startsWith(github.ref, 'refs/tags/')

Use Consistent Structure

Use a consistent structure across your workflow files to make them easier to understand and maintain.

Monitoring and Debugging

Monitoring and debugging your workflows is essential for maintaining their reliability:

Enable Debug Logging

Enable debug logging to get more detailed information about your workflow runs. 

# Set this secret in your repository
# ACTIONS_RUNNER_DEBUG: true
# ACTIONS_STEP_DEBUG: true

Use the GitHub CLI

Use the GitHub CLI to view and debug workflow runs from the command line.

Set Up Notifications

Set up notifications for workflow failures to be alerted when something goes wrong.

  uses: rtCamp/action-slack-notify@v2

Best Practice

Regularly review and update your workflows to ensure they're using the latest best practices and security measures. GitHub Actions is constantly evolving, and new features and improvements are regularly added.

Common Pitfalls and How to Avoid Them

Here are some common pitfalls when using GitHub Actions and how to avoid them:

Not Pinning Action Versions

Problem: Using @main or @master for actions can lead to unexpected behavior if the action is updated.

Solution: Always pin actions to a specific version or SHA.

Hardcoding Secrets

Problem: Hardcoding secrets in workflow files exposes sensitive information.

Solution: Use GitHub Secrets to store sensitive information.

Not Using Caching

Problem: Not using caching can lead to slow workflow runs.

Solution: Use the actions/cache action to cache dependencies and build outputs.

Overusing Matrix Builds

Problem: Using too many matrix combinations can lead to excessive workflow runs and costs.

Solution: Use exclude and include to limit matrix combinations to only what's needed.

Not Handling Failures

Problem: Not handling failures can lead to incomplete workflow runs.

Solution: Use if: failure() or continue-on-error: true to handle failures gracefully.

          python-version: ${{ matrix.python-version }}
          cache: 'pip'
          
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install flake8 pytest
          if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
          pip install -e .
          
      - name: Lint with flake8
        run: |
          flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
          
      - name: Test with pytest
        run: |
          pytest
  
  publish:
    needs: test
    if: github.event_name == 'push' && startsWith(github.ref, 'refs/tags')
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.9'
          
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install build twine
          
      - name: Build and publish
        env:
          TWINE_USERNAME: ${{ secrets.PYPI_USERNAME }}
          TWINE_PASSWORD: ${{ secrets.PYPI_PASSWORD }}
        run: |
          python -m build
          twine upload dist/*

This workflow:

  • Tests the package across multiple Python versions
  • Runs linting checks with flake8
  • Publishes the package to PyPI when a new tag is pushed

Example 5: Terraform Infrastructure Deployment

name: Terraform

on:
  push:
    branches: [ main ]
    paths:
      - 'terraform/**'
  pull_request:
    branches: [ main ]
    paths:
      - 'terraform/**'

jobs:
  terraform:
    name: Terraform
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v2
        with:
          terraform_version: 1.3.0
      
      - name: Terraform Format
        id: fmt
        run: terraform fmt -check
        working-directory: ./terraform
      
      - name: Terraform Init
        id: init
        run: terraform init
        working-directory: ./terraform
      
      - name: Terraform Validate
        id: validate
        run: terraform validate -no-color
        working-directory: ./terraform
      
      - name: Terraform Plan
        id: plan
        if: github.event_name == 'pull_request'
        run: terraform plan -no-color
        working-directory: ./terraform
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      
      - name: Update Pull Request
        uses: actions/github-script@v6
        if: github.event_name == 'pull_request'
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            const output = `#### Terraform Format and Style \`${{ steps.fmt.outcome == 'success' && '' || '' }}
            \`\`\`${{ steps.plan.outputs.stdout }}\`\`\`
            
            *Pushed by: @${{ github.actor }}, Action: \`${{ github.event_name }}\`*`;
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            })
      
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply -auto-approve
        working-directory: ./terraform
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

This workflow:

  • Validates Terraform configurations on pull requests
  • Posts the plan output as a comment on the pull request
  • Applies the Terraform changes when code is pushed to the main branch

Practice Makes Perfect The best way to learn GitHub Actions is by doing. Start with simple workflows and gradually add more complexity as you become more comfortable. Don't be afraid to experiment and learn from your mistakes. Glossary of Terms

Workflow

An automated procedure that you add to your repository, defined in a YAML file in the .github/workflows directory.

Event

A specific activity that triggers a workflow run, such as a push, pull request, or scheduled time.

Job

A set of steps that execute on the same runner. Jobs can run in parallel or sequentially depending on configuration.

Step

An individual task within a job that can run commands, setup tasks, or run an action.

Action

A reusable unit of code that can perform a specific task, such as checking out code or setting up a runtime environment.

Runner

A server that runs your workflows when they're triggered. GitHub provides Linux, Windows, and macOS runners, or you can host your own.

Artifact

Files produced during a workflow run that can be saved and shared between jobs or downloaded after the workflow completes.

Secret

Encrypted environment variables that you create in a repository or organization for use in GitHub Actions workflows.

Environment

A named deployment target with protection rules and secrets that can be referenced in workflow jobs.

Matrix

A set of different configurations for a job, allowing you to test across multiple operating systems, versions, or other variables.

Context

A way to access information about workflow runs, runner environments, jobs, and steps. Each context is an object that contains properties.

Expression

A syntax used in workflows to evaluate variables, contexts, and functions. Expressions are denoted with ${{ expression }}.
